| DEC-001 | Use SQLite (via Python's built-in `sqlite3`, not Django) as the persistence layer | JSON file persistence; Django + its default SQLite | SQLite gives real relational integrity (foreign keys, CHECK constraints) needed to test the Assessmentâ†”Evidence relationship properly, without the overhead of standing up a full Django project for a one-week pilot | REQ-011 (constraint); all AT-* tests run against a reproducible SQLite file created from schema.sql + seed_data.sql |
| DEC-002 | Evidence freshness threshold fixed at 90 days, with 90 itself treated as "not yet stale" (boundary is `> 90`, not `>= 90`) | Threshold configurable per category; 30/60-day threshold | Interview notes gave no explicit number; 90 days was chosen as a reasonable quarterly cadence for infra evidence (dashboards, DR tests) and confirmed as the assumption to validate (A-002). The `> 90` boundary was chosen so that evidence collected "this quarter" is never incorrectly flagged | REQ-004; directly drives TC-005/TC-006 and AT-003's parametrized boundary test |
| DEC-003 | Authentication for the pilot is simulated via a plain username â†’ role lookup table (`user_role`), not real Microsoft/Azure AD SSO | Implement real OAuth/SSO (R8 as originally written) | R8 prescribed a specific identity provider before any requirement defined the actual access-control need, and a full SSO integration is disproportionate for a one-week prototype whose goal is to validate the *role rule*, not the *login mechanism* | REQ-009 (rewritten from R8); explicitly flagged as a known limitation in docs/13; does not affect the correctness of the role-based submission tests (AT-004), since role lookup is mocked at the data layer either way |
| DEC-004 | Split US-002 into US-002 (detect & display missing critical info) and US-002b (recommend next actions), deferring US-002b | Implement AI-generated recommendations directly, per R5 | R5 ("Use AI to generate recommendations") had no defined logic or data source for recommendations, and mixed a solution technology into the requirement (see P-005). Splitting lets the testable core (missing-info detection) ship and be verified now, while recommendation logic is properly elicited later | US-005 story split section; REQ-003 implemented and tested (AT-001/AT-002), recommendation feature intentionally out of app scope for this exam |
| DEC-005 | Stale evidence is FLAGGED, never auto-rejected or blocking of submission | Block submission entirely if any evidence is stale | Initial assumption (pre-change-request) was ambiguous on this point; the mid-week change request explicitly clarified this ("evidence older than 90 days must be flagged, not rejected"), which was then implemented as a non-blocking flag in `readiness_rules.assessment_readiness()` and `submit_assessment()` | REQ-004 wording updated after CR; AT-003 and AT-004 confirm staleness never affects the `ready`/`success` boolean, only the `stale` list |
| DEC-006 | Vibe Coding app scope = Option A (Evidence validation slice) extended with the REQ-005 role rule, rather than pure Option A or pure Option C | Build the full Option B readiness checklist as well | The interview notes and traceability chain both center on evidence (source/owner/freshness) AND the role-based submission control together (Security Officer's quote directly couples both concerns: "Only approved AMS leads should change readiness assessments. Evidence must be traceable."), so combining a thin slice of Option A + the specific role rule from Option C gives the most coherent, testable vertical slice without building the full checklist UI (Option B), which was judged out of scope for the one-week timebox | docs/11_vibe_coding_app.md; app/main.py implements both evidence CRUD and role-gated submission; readiness checklist (fixed 5 categories) is still enforced via the `readiness_question` table even though there is no separate "checklist" screen |
| DEC-007 | Automated tests use a dedicated `tests/test_readiness.db`, created and destroyed per test run, fully separate from `app/readiness.db` used for manual demoing | Reuse the same `app/readiness.db` for both app demo and tests | Prevents test runs from corrupting/resetting the manually-demoed data, and guarantees test reproducibility regardless of what has been clicked in the running app | docs/12_automated_tests.md; tests/test_readiness_database.py `conn` fixture |
| DEC-008 *(CR)* | Model the RFC tool as a single Q&A pair per RFC record (one `question` + one `answer_text`), not a full multi-message threaded chat, with a lightweight `is_faq_candidate` boolean instead of a separate FAQ table | Full threaded conversation model (multiple messages per RFC); a separate `Faq` entity populated automatically | The change request explicitly says the tool should NOT be "just a chat" but also doesn't require full threading; a single structured Q&A already satisfies "structured technical documentation" and avoids the complexity of a message-thread model or a separate FAQ store for a one-week pilot. `is_faq_candidate` gives a cheap hook toward "may contribute to future FAQs" without building an actual FAQ page | app/schema.sql (`rfc` table); app/rfc_rules.py; AT-005..AT-008; docs/10_data_architecture.md |
| DEC-009 *(CR)* | Only role `TransitionLead` may raise an RFC; only `Contributor` or `TransitionLead` may answer one (SecurityOfficer/AMSManager excluded from both actions in this pilot) | Allow any recognized role to raise/answer | The CR frames the RFC explicitly as something the Transition Lead requests from a Contributor; allowing TransitionLead to also answer supports self-documentation edge cases (e.g. clarifying their own question) without loosening the rule to all roles, consistent with the least-privilege pattern already used for submission (REQ-005/REQ-009) | app/rfc_rules.py (`can_raise_rfc`, `can_answer_rfc`); AT-006, AT-008 |
| DEC-010 *(CR)* | Keep `RFC` as an entity independent from `Evidence`, rather than reusing the `evidence` table with a new category value | Add an `is_rfc` flag or a "RFC" category to the existing `evidence` table | `Evidence` represents proof of something that already exists (with a freshness date that ages), while `RFC` represents an open request-and-answer workflow with its own lifecycle (open/answered) and different actors (`raised_by`/`answered_by`); conflating them would incorrectly apply the evidence-freshness rule (REQ-004) to open questions, which has no meaningful "freshness" | app/schema.sql (separate `rfc` table); docs/10_data_architecture.md |
